# Quick Task 260521-l5c — Fix CR-04

## Objective

Close CR-04 from `02-REVIEW.md` and `02-VERIFICATION.md` anti-patterns row 4: `updateCareUnitMedication` makes two separate top-level `prisma.update` calls — one for `CareUnitMedication` (stock/threshold), one for `Medication` (name/atcCode/form/strength on user-source rows). If the second call fails after the first commits, the row is left half-updated with no rollback.

## Tasks

### Task 1: Wrap both updates in $transaction

- **File:** `apps/api/src/services/medication.service.ts` (lines 446-465)
- **Approach:** Use `prisma.$transaction(async (tx) => { ... })` callback form to match the style already used by `createCareUnitMedication` (line 312). Both updates run inside one transaction; partial failure rolls back atomically.
- **Preserve:** Existing short-circuit (`hasCumUpdate`/`hasMedUpdate` guards), defensive scope re-check (`updatedRow.careUnitId !== careUnitId`), and `toListItem` return path remain unchanged.
- **Diff scope:** ~24 lines, no API contract change, no caller change.

## Must Haves

- `prisma.medication.update` and `prisma.careUnitMedication.update` in `updateCareUnitMedication` both execute via `tx` inside one `prisma.$transaction(async (tx) => {...})`.
