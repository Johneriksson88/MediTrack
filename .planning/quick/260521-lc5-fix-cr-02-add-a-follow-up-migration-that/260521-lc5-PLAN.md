# Quick Task 260521-lc5 — Fix CR-02

## Objective

Close CR-02 from `02-REVIEW.md` and `02-VERIFICATION.md` anti-patterns row 3: the GIN trigram index created by the Phase 2 migration is on `lower("name")`, but Prisma's `{ name: { contains: q, mode: 'insensitive' } }` emits `"name" ILIKE '%q%'` with the bare column on the LHS. Postgres' planner can't use a functional index on `lower("name")` for a predicate on `"name"`, so every name search degrades to a sequential scan over ~43k Medication rows.

## Approach Choice

Two ways to fix:

1. **Edit the original migration** (`20260521000000_medication_catalog/migration.sql`) — single-line change but **never edit a committed migration** in Prisma: it breaks anyone who already ran the original (Prisma will detect schema drift on next `migrate deploy`).
2. **Add a new follow-up migration** that drops the wrong index and creates the aligned one — keeps migration history honest; fresh installs run both in sequence (slightly wasteful one-time cost; correct long-term).

We picked option 2. Standard pattern for post-deploy index corrections.

## Tasks

### Task 1: Add new migration

- **File:** `apps/api/prisma/migrations/20260521130000_align_medication_name_gin_index_cr02/migration.sql` (new)
- **Operations:**
  - `DROP INDEX IF EXISTS "Medication_name_lower_trgm_idx";`
  - `CREATE INDEX "Medication_name_trgm_idx" ON "Medication" USING gin ("name" gin_trgm_ops);`
- **Why bare column instead of lower():** pg_trgm's `gin_trgm_ops` operator class natively supports ILIKE (since PostgreSQL 9.1) — the trigram matching is case-insensitive against the indexed column. A plain `gin("name" gin_trgm_ops)` index is exactly what Prisma's emitted ILIKE needs.
- **Why `IF EXISTS`:** Defensive — makes the migration idempotent against environments that somehow didn't apply the original (e.g., a partial state during local dev iteration).

### Task 2: Apply and verify

- Run `pnpm --filter @meditrack/api exec prisma migrate deploy` against the running dev Postgres.
- Run `\d+ "Medication"` to confirm only the new `Medication_name_trgm_idx` exists (and the old `_lower_trgm_idx` is gone).
- Run `EXPLAIN SELECT id, name FROM "Medication" WHERE name ILIKE '%alved%' LIMIT 20;` and verify the plan shows `Bitmap Index Scan on Medication_name_trgm_idx` rather than `Seq Scan on Medication`.

## Must Haves

- Migration file present with DROP + CREATE statements.
- `prisma migrate deploy` succeeds against the dev DB (one new migration applied, no schema drift).
- `EXPLAIN` of `name ILIKE` shows the new index in use.
- No change to `schema.prisma` (custom GIN/trgm indexes are migration-only — they don't have a Prisma schema representation).
- `pnpm --filter @meditrack/api build` exit 0 (sanity check that no other expectations on the old index name broke).

## Out of Scope

- `Medication.atcCode` index optimization — separate concern; ATC filter uses `startsWith` which a btree on the column handles fine (and `@@index([atcCode])` already exists in schema.prisma).
- Search ranking by trgm similarity — defer to a phase if surfaced.
- Updating prisma to v7 — version-upgrade noise, not in this task's scope.
