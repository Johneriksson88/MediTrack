## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser DOM → user input | Form inputs, route URLs, click handlers — sanitized by React's default escaping |
| FE permission gates (`<Can>`, `RoleRoute`) → user perception | Defense-in-depth ONLY (AUTH-06); the BE is the real security boundary |
| Logout flow → session cookie | DELETE /api/auth/session is the only state-changing request from this plan |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | Logout button visible to unauthenticated users | accept | Logout button only renders under `<AuthGate>` (authenticated routes), so it's never visible to unauthenticated users. Acceptable. |
| T-04-02 | Tampering | User crafts URL to access /admin/audit as non-admin | mitigate | `<RoleRoute roles={['admin']}>` redirects non-admin to `/dashboard`. Defense-in-depth only — Phase 5 will add the real BE enforcement on `/api/audit/*`. The page is a stub in Phase 1. |
| T-04-03 | Information Disclosure | Admin nav item visible to non-admin (config leak) | mitigate | Sidebar + BottomTabBar filter the admin item out when `user.role !== 'admin'`. Defense in depth only (AUTH-06). |
| T-04-04 | Spoofing | CSRF on DELETE /api/auth/session | mitigate | `SameSite=Lax` cookie blocks third-party form-POST CSRF (Plan 02 mitigation). Cross-origin Phase 7 follow-up documented in CONTEXT.md Deferred Ideas. |
| T-04-05 | Information Disclosure | XSS via user-controlled DOM content | accept | All rendered values (user.name, user.careUnit.name) come from the BE which validates inputs via Zod. React's default escaping prevents script injection. No `dangerouslySetInnerHTML` anywhere in Plan 04. |
| T-04-06 | Elevation of Privilege | FE `<Can>` bypass via DevTools | accept | AUTH-06 explicitly says the FE is defense-in-depth, not the security boundary. A user editing local state to render the admin button can still only succeed if the BE permits the action (Plan 03's `requirePermission` enforces 403). Acceptable by design. |
| T-04-07 | Denial of Service | Spamming logout endpoint | accept | Logout is idempotent (Plan 02 spec: DELETE returns 204 even with no cookie). No DoS risk beyond normal request volume. |
| T-04-SC | Tampering | shadcn add popover (CLI install) | mitigate | shadcn registry is `ui.shadcn.com` (UI-SPEC §Registry Safety — official). Underlying packages: `@radix-ui/react-popover` (top-100 npm). No `[ASSUMED]` / `[SUS]` / `[SLOP]` packages introduced. |
