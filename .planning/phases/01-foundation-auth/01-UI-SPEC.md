## Spacing Scale

Base unit: 4px (Tailwind default). All layout values must be multiples of 4.

| Token | px | Tailwind class | Primary usage in Phase 1 |
|-------|----|----------------|--------------------------|
| 1 | 4px | `p-1` / `gap-1` | Icon-label gap in sidebar nav items |
| 2 | 8px | `p-2` / `gap-2` | Compact element spacing, badge inner padding |
| 3 | 12px | `p-3` | Sidebar icon button padding (icon-only at md) — project extension, see note |
| 4 | 16px | `p-4` / `gap-4` | Default card padding, form field gaps, login form internals |
| 5 | 20px | `p-5` | Login card vertical internal padding step — project extension, see note |
| 6 | 24px | `p-6` / `gap-6` | Login card padding (`p-8` used — see exception), section spacing |
| 8 | 32px | `p-8` | Login card padding at ≥480px viewport |
| 12 | 48px | `py-12` | Vertical centering breathing room on login page |

**Standard set:** 4, 8, 16, 24, 32, 48, 64 (Tailwind base scale subset).
**Project extensions:** 12px (`p-3`) — used for icon-button padding at md breakpoint where 8px crowds the icon and 16px breaks visual rhythm with adjacent components. 20px (`p-5`) — used as the login card internal vertical padding step between `p-4` (cramped on mobile) and `p-6` (too generous at 360px). Both extensions are intentional, scoped, and used only at the noted surfaces.

**Exceptions:**
- Login card: `p-6` at ≤479px, `p-8` at ≥480px (keeps card content reachable on narrow phones without cramping on larger screens).
- Touch targets: all interactive nav elements (bottom tab bar items, sidebar icon buttons) must meet **44×44px minimum** tap target. Use `min-h-[44px] min-w-[44px]` or equivalent. This overrides compact spacing where necessary.
- Bottom tab bar height: **56px** plus `env(safe-area-inset-bottom)` for iOS home indicator clearance. Use `pb-[env(safe-area-inset-bottom)]` on the bar itself.

## Typography

Font family: `font-sans` (Tailwind default system stack — see Design System above).

**Weights (2):** `font-semibold` (600) — headings, badges, captions, nav labels, interactive labels. `font-normal` (400) — body text, form content, descriptive text. No third weight is permitted in Phase 1.

| Role | Tailwind size token | px equiv | Weight | Tailwind weight | Line height | Usage |
|------|---------------------|----------|--------|-----------------|-------------|-------|
| Display / Page heading | `text-2xl` | 24px | Semibold | `font-semibold` (600) | `leading-tight` (1.25) | Login page title (`Logga in`), stub page headings |
| Section heading | `text-xl` | 20px | Semibold | `font-semibold` (600) | `leading-snug` (1.375) | Empty state card heading (e.g. `Läkemedel`), `Konto` page heading |
| Body / default | `text-sm` | 14px | Regular | `font-normal` (400) | `leading-relaxed` (1.625) | Form labels, input values, nav item labels, descriptive text |
| Small / caption | `text-xs` | 12px | Semibold | `font-semibold` (600) | `leading-normal` (1.5) | Role badges, status chips, helper text, muted captions |

**Rule: use exactly these four size levels in Phase 1. Do not introduce `text-base` (16px) or `text-lg` (18px) in Phase 1 surfaces — they will appear in later phases for data-dense tables.**

No monospace / code typography required in Phase 1.

### Border

All borders: `border border-[#E2E8F0]` (or `border-border` if using shadcn CSS vars). Border radius follows shadcn new-york preset: `rounded-lg` for cards, `rounded-md` for inputs and buttons, `rounded-full` for badges.

### Focus Ring Specification

All interactive elements (buttons, inputs, links, nav items, tab bar items):

```
outline: none;
ring: 2px solid #2563EB;
ring-offset: 2px;
```

Tailwind: `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-[#2563EB] focus-visible:ring-offset-2`

Use `focus-visible` (not `focus`) so mouse users don't see the ring. Do not suppress focus rings.

### Disabled State

Disabled interactive elements: `opacity-50 cursor-not-allowed`. Do not use a different color; just reduce opacity. This applies to the login submit button during form submission (`isSubmitting`).

### Focus Order & Keyboard Behavior

- Tab order follows DOM order: top bar → sidebar (md+) → main content.
- `Escape` closes the user pill popover (desktop).
- `Enter` / `Space` submits the login form when focus is on the submit button.
- `Enter` activates nav links when focused.
- All nav items, buttons, inputs, and the user pill popover trigger are reachable by keyboard alone.
- No keyboard shortcuts beyond standard browser/ARIA patterns in Phase 1.

### Back-Button Behavior

- After successful login: `navigate(from ?? '/', { replace: true })` — the login page is replaced in history so the back button does not return to login while authenticated.
- 401 from `/me`: `<Navigate to="/login" state={{ from: location.pathname }} replace />` — preserves the intended destination for post-login redirect.

## Components & States

### 1. Login Page (`/login`)

**Components used:** shadcn `Card`, `CardHeader`, `CardContent`, `CardFooter`, `Label`, `Input`, `Button`, `Alert`.

**Layout within card:**
```
CardHeader
  └─ page title: "Logga in" (text-2xl font-semibold)
CardContent
  └─ form
       ├─ [Alert — destructive, hidden until error]
       ├─ Label + Input (e-post)
       ├─ Label + Input (lösenord, type=password)
       └─ Button (primary, full-width, type=submit)
```

**States:**

| State | Visual |
|-------|--------|
| Idle | Card visible, inputs empty, button active, no Alert |
| Validating (client) | Inputs show `aria-invalid="true"` + red border (`border-destructive`) on invalid fields; no Alert yet |
| Submitting | Button disabled + `opacity-50` + spinner (`Loader2` icon with `animate-spin`) replacing button text; inputs disabled |
| Server error | Alert variant=destructive visible immediately above the Button; text: `Fel e-post eller lösenord.`; Button re-enabled; inputs re-enabled and retain values |
| Success / redirect | AuthGate takes over; login page unmounts; navigate to `from` or `/dashboard` |

**Error accessibility:** `<Alert>` has `role="alert"` (shadcn default). Input with server error: `aria-invalid="true"` + `aria-describedby` pointing to the Alert's id.

**No success toast.** Navigation is the success signal.

### 8. 403 / `<Can>` Gate Pattern (Phase 1 demo)

The `admin:ping` button on the Konto page demonstrates 403 handling. Locked pattern:

- Render the button for all roles using `<Can action="admin:ping">` — but the component renders the button regardless and relies on server-side 403 to demonstrate the flow.

  **Correction to `<Can>` pattern:** `<Can action="admin:ping">` renders children only when `can('admin:ping')` is true (i.e., user is admin). For non-admin users, `<Can>` renders nothing — the UI gate is defense-in-depth.

  To demo 403 for reviewers, the Konto page shows:
  - Admin users: `<Can action="admin:ping"><Button>Admin ping</Button></Can>` — button visible and functional.
  - Non-admin: The button is hidden. A static note in muted text: `Denna åtgärd kräver adminrättigheter.` (`text-xs text-[#64748B]`) — demonstrates the gate visually without requiring the user to trigger a 403.

  The backend still enforces the role check. The frontend gate (hiding the button) is the defense-in-depth layer per AUTH-06.

- **If a non-admin somehow triggers the API and receives 403:** Display shadcn `Alert` variant=destructive inline with text: `Du saknar behörighet att utföra denna åtgärd.` — no toast.

### 9. Inline Error Pattern (form submissions)

shadcn `Alert` variant=destructive placed **immediately above the submit button**.

```jsx
{serverError && (
  <Alert variant="destructive" role="alert" id="form-error">
    <AlertDescription>{serverError}</AlertDescription>
  </Alert>
)}
<Button type="submit" ...>...</Button>
```

- The errored input(s) get `aria-invalid="true"` and `aria-describedby="form-error"`.
- Toast (`sonner` or shadcn toast) is reserved for **transient success/info** notifications only. None appear in Phase 1.

### 10. 401 Redirect

No UI. When `useQuery(['me'])` returns a 401 response:
```jsx
<Navigate to="/login" state={{ from: location.pathname }} replace />
```
No toast. No alert. The login page is the feedback.

## Accessibility Floor

Every item below is a hard requirement, not a best-effort guideline.

| Requirement | Implementation |
|-------------|----------------|
| Keyboard reachability | All interactive elements in natural tab order; no keyboard traps |
| Visible focus ring | `focus-visible:ring-2 focus-visible:ring-[#2563EB] focus-visible:ring-offset-2` on every interactive element |
| Icon-only sidebar (md) | `aria-label` on every `<NavLink>` |
| Bottom tab bar | `<nav aria-label="Primary">` wrapping the bar; `aria-current="page"` on active tab |
| Active nav indicator | `aria-current="page"` on active link (not just visual styling) |
| Form errors | `aria-invalid="true"` on errored inputs + `aria-describedby` → Alert `id` |
| Login Alert | `role="alert"` (shadcn default on Alert variant=destructive) |
| Color contrast | All text/bg pairs verified above ≥ 4.5:1 (see Color section) |
| Language declaration | `<html lang="sv">` in `index.html` |
| Touch targets | All tap targets ≥ 44×44px (nav items, buttons, tab bar items) |
| Safe area inset | Bottom tab bar uses `env(safe-area-inset-bottom)` |
| Escape key | Closes user pill popover (Radix Popover handles this natively) |

## No-Horizontal-Scroll Guarantee

Verified requirements per success criterion 3:

| Breakpoint | Guarantee | Implementation |
|-----------|-----------|----------------|
| 360px | Login card: `mx-4` gives 16px margin each side; card = 328px max. No overflow. | `max-w-[28rem] w-full mx-4` |
| 360px | App shell: sidebar hidden (`hidden md:flex`); bottom tab bar full width. Main content `w-full overflow-x-hidden`. | Flex column layout, no fixed-width children |
| 768px | Sidebar 64px icon-only; main content `flex-1 min-w-0` (prevents flex child blowout). | `min-w-0` on main content flex child |
| 1024px | Sidebar 240px; main content fills remainder. Container `max-w-7xl mx-auto` inside. | `flex flex-row`, sidebar `w-60`, content `flex-1 min-w-0` |
| 1440px | Same as 1024px. Content area max 1200px (1440 − 240 sidebar = 1200). `max-w-7xl` = 1280px; with `px-8` the usable area is ~1184px. No overflow. | CSS max-width constraint |

**Rule:** Never use `w-screen` inside a flex child. All full-width elements use `w-full`. All flex children that must not overflow use `min-w-0`.
