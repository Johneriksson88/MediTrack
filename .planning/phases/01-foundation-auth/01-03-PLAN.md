<objective>
Land the RBAC layer end-to-end on top of the Walking Skeleton from Plan 02:
- Server-side enforcement via `requirePermission(action: ActionKey)` Fastify preHandler factory and the centralized `PERMISSIONS` map (D-15).
- A real demo endpoint `GET /api/admin/ping` that returns 200 to admin, 403 to others, 401 with no session — the three paths Phase 1 success criterion #2 demands.
- `/me` now returns a computed `permissions: ActionKey[]` field (D-18) so the FE knows what the user can do without a second round-trip.
- Frontend defense-in-depth primitives: `useAuth()`, `useCan(action)`, `<Can action="...">` — Pattern L — backed by the same `useQuery(['me'])` source of truth (D-17).

Purpose: Reviewer's §6 question "how would you retrofit auth" is answered by *doing* it: the BE is the security boundary; the FE primitives are defense in depth (AUTH-06). Phase 2+ extends `PERMISSIONS` (`medication:create`, `order:confirm`, etc.) without touching this scaffolding.

Output: `admin:ping` action key wired in shared + API + tests; FE `useAuth`/`useCan`/`<Can>` primitives ready for Plan 04 to render the Konto admin gate note.
</objective>
