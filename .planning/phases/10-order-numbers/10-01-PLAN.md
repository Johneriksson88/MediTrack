## Phase Goal

**As a** nurse, apotekare, or admin using MediTrack, **I want to** see every order labelled with a unique, human-readable order number that persists through its full lifecycle, **so that** I can reference an order conversationally (in a hand-off, in an audit lookup, in a phone call) without quoting an opaque cuid.

<objective>
Land the backend end-to-end for Phase 10 ORD-11: schema migration (new Order columns + new OrderNumberCounter model + UNIQUE constraint), backfill of every existing row, runtime mint inside the createDraftOrder $transaction with row-level write-lock semantics (D-160, §6 concurrency answer reused from Phase 4 STK-02), shared formatOrderNumber utility (single source of truth per D-165), and contract widening so every order-rendering surface receives orderNumber as a required string over the wire. Slice 1 ships BE end-to-end; FE still shows raw id-substring until Slice 2 (Plan 02) lands.

Purpose: Make the order number a stable, server-authoritative, audit-covered identifier before any FE consumer learns to render it — Plan 02 cannot fail open because the wire contract enforces presence at typecheck and Zod-parse boundaries.
Output: Migration 0013_order_numbers applied; 5-scenario integration suite green; all existing order/dashboard integration tests extended with orderNumber assertions; audit log auto-captures the new fields via the Phase 5 $extends middleware.
</objective>
