<objective>
Tighten the audit-log enforcement around three known gaps surfaced in 05-REVIEWS.md:

- **HIGH #4 — createMany is unaudited and unbanned.** Plan 01 D-93 deliberately skipped intercepting `createMany` because "only seed uses it." But ESLint doesn't enforce that constraint; a future contributor could call `prisma.medication.createMany([...])` in a service and bypass the audit middleware entirely. This plan extends the existing `no-restricted-syntax` rule to ban `*.createMany` outside `apps/api/prisma/seed.ts`.
- **MEDIUM #5 — `$executeRaw` / `$executeRawUnsafe` bypass.** Plan 01 D-93 acknowledged that the audit extension cannot see raw queries; the README disclaims this as a v2 gap. This plan ships the v1 closure: a CI grep test asserting zero off-allowlist matches. The allowlist contains one entry today (Phase 4's `FOR UPDATE` lock acquisition — a READ, not a write).
- **LOW #12 — empty-string entityId sentinel.** Plan 05-05 (WR-07) split the `auth.login_failed` path into entityType='auth_attempt' (unknown-email) and entityType='session' (known-user). The old code wrote `entityId = ''` for the unknown-email branch; that pattern is now closed in the application code. This plan adds a BEFORE INSERT trigger as a backstop — no audit row can carry an empty-string entityId regardless of which code path writes it.

The three findings cluster: all three are DEFENSE-IN-DEPTH guards at the boundary the existing append-only story doesn't yet cover. Bundling them into one plan keeps the additions atomic and the README narrative coherent.

Purpose:

- **createMany ban (HIGH #4).** Extend the existing `.eslintrc.cjs no-restricted-syntax` rule with a selector that matches `*.createMany` member expressions. The seed file `apps/api/prisma/seed.ts` is exempted via an ESLint `overrides` block. The rule message cites D-93 and 05-REVIEWS.md HIGH #4 so the lint error surface explains itself.
- **$executeRaw CI grep (MEDIUM #5).** Add a new integration test (Test 15) that runs `git grep` for `prisma.\$executeRaw|prisma.\$executeRawUnsafe` across `apps/` and `packages/` and asserts every match is in a documented allowlist. <why_this_approach>

**createMany via ESLint, not via Prisma extension extension (HIGH #4).** Two options were considered:
- (a) Extend the audit Prisma extension to ALSO intercept `createMany`: load matching rows by PK after the op (since the op returns `{count: N}`, not the rows themselves), emit N audit rows. This is the more complete fix but adds non-trivial complexity to the extension (handling the PK-load-after pattern, deciding which fields to allowlist, etc.) and introduces a new failure surface (what if the PK load fails?).
- (b) Ban `createMany` via ESLint, with seed.ts as the documented exception. Simpler; aligns with D-93's stated decision. Aligned with 05-REVIEWS.md's recommended fix.

This plan ships (b). Rationale: D-93 is the locked decision; the architectural choice is to NOT intercept createMany. The ESLint ban operationalises that decision so future contributors don't accidentally use createMany in a service. If a future phase needs bulk-import (Phase 6's AI categorization might propose this for the NPL re-categorize feature), that phase reopens D-93 and chooses option (a) consciously.

Why NOT also include `upsert` in the ban explicitly: the existing rule already bans `prisma.auditEvent.upsert` (D-99), but it allows `upsert` on OTHER models (which is fine — upsert IS intercepted by the extension as a logical create-or-update pair). The createMany ban targets ONLY createMany on any model except seed.ts.

**$executeRaw CI grep, not a Prisma extension extension (MEDIUM #5).** The same option analysis: the extension cannot see raw queries (Prisma's `$extends({query})` only intercepts model-method calls). The only ways to enforce "no raw writes" are:
- (a) CI grep with allowlist (this plan).
- (b) Wrap `prisma.$executeRaw` with a runtime guard that throws unless the call site is allowlisted. Adds runtime overhead + complexity; doesn't catch the issue at PR time.

This plan ships (a). The grep is fast (sub-second), runs in CI, and gives a clear PR-time signal.

Why the allowlist is hardcoded in the test file: simpler than a separate config file. The allowlist is small (one entry today; expected to stay small) and any addition is a PR-level decision. Keeping it inline with the test makes the decision visible.

**BEFORE INSERT trigger for empty entity_id (LOW #12).** A check constraint would also work: `CHECK (entity_id <> '')`. Either works for the data-integrity contract. The BEFORE INSERT trigger is chosen for symmetry with the existing 0008 trigger guards (which already use BEFORE triggers + RAISE EXCEPTION); future contributors reading the migration history see a consistent pattern. The check constraint would be slightly faster at write time (it's evaluated as part of the row INSERT, not as a separate trigger fire), but the audit row INSERT is not on any hot path — the latency difference is unmeasurable.

Why NOT enforce this in the Prisma extension (e.g., refuse to write a row with entityId === ''): Plan 05-05 (WR-07) already did this in the application code (the auth-attempt branch sets entityId to the attempted email, never ''). The trigger is the DB-layer backstop for the same contract, mirroring the 0008 trigger / REVOKE pattern.

**Bundling.** The three findings are independent in scope but share a single section in README ("the audit log's defense-in-depth guards"), and the new test + migration both touch the audit pipeline. Splitting into three plans would create three adjacent PRs/commits with overlapping diffs in README and audit.integration.test.ts; bundling keeps the work atomic.

</why_this_approach>

```typescript
import { execFileSync } from 'node:child_process';

it('CI grep: no off-allowlist $executeRaw matches in apps/ or packages/ (Phase 5 MEDIUM #5)', async () => {
  // ALLOWLIST: file paths permitted to use $executeRaw* (with rationale).
  const ALLOWLIST: ReadonlyArray<{ file: string; reason: string }> = [
    {
      file: 'apps/api/src/services/order.service.ts',
      reason: 'Phase 4 D-79 — SELECT ... FOR UPDATE lock acquisition (READ, not WRITE)',
    },
  ];

  // Run git grep from the repo root.
  const output = execFileSync(
    'git',
    ['grep', '-nE', 'prisma\\.\\$executeRaw(Unsafe)?\\b', 'apps', 'packages'],
    { encoding: 'utf8', cwd: process.cwd() },
  ).trim();

  // git grep exits 1 with no output when there are no matches; child_process throws
  // in that case. Wrap to handle the no-matches case as "success" (no work needed).
  const matches = output.split('\n').filter((l) => l.length > 0);

  // Every match must be in the allowlist.
  const allowlistFiles = new Set(ALLOWLIST.map((e) => e.file));
  const offAllowlist = matches.filter((m) => {
    const path = m.split(':')[0]; // git grep output format: "path:line:content"
    return !allowlistFiles.has(path);
  });

  if (offAllowlist.length > 0) {
    // Fail loudly with the offending lines so PR review sees the diff.
    throw new Error(
      `Off-allowlist $executeRaw matches found:\n${offAllowlist.join('\n')}\n\nAdd the file to the ALLOWLIST in this test with a reason, or refactor to use a Prisma model method (which IS intercepted by the audit extension).`,
    );
  }
});
```

```sql
-- Phase 5 — BEFORE INSERT trigger rejecting empty entity_id on AuditEvent
-- (05-REVIEWS.md LOW #12 — DB-layer backstop for the WR-07 sentinel-empty
-- class that Plan 05-05 closed at the application code layer).
--
-- The Plan 05-05 fix split auth.login_failed entityType so the unknown-email
-- branch carries entityType='auth_attempt' with entityId = <attempted email>
-- instead of entityId = ''. That fix is in code today (auth.service.ts).
-- This trigger is the DB-layer backstop: even if a future code path forgets
-- to set entityId, the DB rejects the row with SQLSTATE 23514.

CREATE OR REPLACE FUNCTION "AuditEvent_reject_empty_entity_id"()
RETURNS trigger
LANGUAGE plpgsql
AS $$
BEGIN
  IF NEW.entity_id = '' OR NEW.entity_id IS NULL THEN
    RAISE EXCEPTION 'audit_events.entity_id must be a non-empty string (Phase 5 LOW #12 backstop)'
      USING ERRCODE = '23514';
  END IF;
  RETURN NEW;
END;
$$;

DROP TRIGGER IF EXISTS "AuditEvent_reject_empty_entity_id" ON "AuditEvent";
CREATE TRIGGER "AuditEvent_reject_empty_entity_id"
  BEFORE INSERT ON "AuditEvent"
  FOR EACH ROW
  EXECUTE FUNCTION "AuditEvent_reject_empty_entity_id"();
```
