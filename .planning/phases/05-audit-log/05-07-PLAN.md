<objective>

The original Plan 01 D-98 design was REVOKE … FROM CURRENT_USER. Plan 01's SUMMARY acknowledges this turned out to be a no-op because the runtime role OWNED the AuditEvent table — Postgres bypasses GRANT/REVOKE for owners. Plan 01 worked around it by adding BEFORE-triggers (the owner-binding guard); REVOKE was kept as defense-in-depth for a future non-owner role.

This plan operationalises that future: a NAMED non-owner role `meditrack_app` exists from day 1, the application connects as that role, and the REVOKE is bound to the name. The trigger guard (Plan 01) stays as the owner-side fallback; the two layers compose properly.

<why_this_approach>

**Named role over CURRENT_USER (HIGH #3).** Plan 01's REVOKE FROM CURRENT_USER was captured at migration runtime. The `meditrack` role that ran the migration also owned the table — REVOKE was a no-op. Plan 01 added triggers (Layer 2a) to bind owners. This plan adds the NAMED-role REVOKE (Layer 2b) so the original D-98 contract ("Postgres rejects the write at the role layer") is met BOTH for owners (via triggers) AND for the actual runtime app role (via REVOKE).

The two layers compose: queries from the table-owner role hit the trigger first (RAISE EXCEPTION SQLSTATE 42501); queries from the named app role hit the REVOKE first (the grant check fails before the trigger fires). Either layer is sufficient on its own for its respective role; the layering is defense-in-depth.

Why NOT delete the trigger (Plan 01's Layer 2a) and rely only on the named role's REVOKE: the trigger remains useful for any one-off admin connection that lands as the owner (e.g., a developer running `psql` as `meditrack` to inspect the database). The trigger guards against accidental writes from such an admin connection. Cost of keeping it: ~10 lines of plpgsql + one trigger per DML class. Negligible.

Why NOT create the named role via a docker-compose initdb script (alternative to a Prisma migration): Prisma migrations are the single-source-of-truth for the database's structure in this project. The role is part of the database's structure (it's referenced by every GRANT statement). Putting role creation in a migration means a fresh `prisma migrate deploy` brings up the same role state on every environment — production, CI, local dev — without depending on docker-compose-specific initdb shell scripts. The downside: migrations run AS the owner, so creating a role and granting it privileges is the migration's responsibility. Postgres allows this via CREATE ROLE + GRANT … TO meditrack_app statements; idempotent via IF NOT EXISTS.

**Runtime role swap via docker-compose.yml (the operational change).** Two env vars:
- `DATABASE_URL=postgresql://meditrack_app:${POSTGRES_APP_PASSWORD}@postgres:5432/meditrack` — runtime queries from the api service.
- `DIRECT_URL=postgresql://meditrack:${POSTGRES_PASSWORD}@postgres:5432/meditrack` — migration runner (prisma migrate deploy), seed scripts. Prisma already documents the DATABASE_URL vs DIRECT_URL distinction for pooler vs direct connections; we use the same split for owner vs app role.

The `apps/api/prisma/schema.prisma` file's datasource block needs a `directUrl = env("DIRECT_URL")` line if not already present; this is the documented Prisma pattern.

Why NOT use a single connection string with `?application_name=meditrack_app` (Postgres allows the app to declare its name without role-switching): application_name is observational metadata only; Postgres still uses the connection role for GRANT/REVOKE checks. We need an actual role swap.

Why NOT use SET ROLE inside the application code at query time (postgres-side role-switching mid-connection): adds runtime overhead on every Prisma query; Prisma doesn't have a documented SET ROLE hook; complexity outweighs the value of "no extra DB role." A separate role for runtime queries is the conventional pattern.

**Idempotent migration.** The migration uses three patterns:
- `DO $$ ... $$ EXCEPTION WHEN duplicate_object THEN NULL` for CREATE ROLE.
- `GRANT ... TO meditrack_app` is idempotent natively (re-running grants the same privilege; no-op).
- `REVOKE ... FROM meditrack_app` is idempotent natively (re-running REVOKE on an already-revoked privilege is a no-op).

So `prisma migrate deploy` is safe to re-run; existing databases with migration 0010 already applied see no schema change.

**Test coverage.** Plan 05-03's Test 4 (`prisma.$executeRawUnsafe("UPDATE \"AuditEvent\" SET ...")` rejects with permission denied) should continue to pass — the runtime role is now `meditrack_app` (REVOKE binds), not `meditrack` (trigger binds), so the rejection path is different but the observable behavior is identical. Optionally extend Test 4 to assert `SELECT current_user` returns `meditrack_app` (proving the role swap landed) and to additionally test that the OWNER `meditrack` ALSO can't UPDATE (proving the trigger guard still works against admin connections). This extension is in Task 4.

</why_this_approach>

POST-PLAN migration 0010 shape (apps/api/prisma/migrations/20260523000000_0010_audit_events_named_app_role/migration.sql):

```sql
-- Phase 5 — Named app role for audit-events append-only enforcement
-- (05-REVIEWS.md HIGH #3 + the 05-01 SUMMARY's "owner REVOKE was no-op" finding).
--
-- WHY THIS MIGRATION EXISTS ...
--
-- The migration body uses three statements (idempotent via DO block + IF NOT EXISTS):

DO $$
BEGIN
  CREATE ROLE meditrack_app WITH LOGIN PASSWORD 'meditrack_app_dev';
EXCEPTION
  WHEN duplicate_object THEN
    -- Role already exists; ensure its login + password match the expected values.
    ALTER ROLE meditrack_app WITH LOGIN PASSWORD 'meditrack_app_dev';
END $$;

GRANT CONNECT ON DATABASE meditrack TO meditrack_app;
GRANT USAGE ON SCHEMA public TO meditrack_app;
GRANT SELECT, INSERT, UPDATE, DELETE, REFERENCES ON ALL TABLES IN SCHEMA public TO meditrack_app;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO meditrack_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE, REFERENCES ON TABLES TO meditrack_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT USAGE, SELECT ON SEQUENCES TO meditrack_app;

-- The hardening: REVOKE UPDATE, DELETE, TRUNCATE on AuditEvent FROM meditrack_app
-- so the app role physically cannot mutate audit rows.
REVOKE UPDATE, DELETE, TRUNCATE ON "AuditEvent" FROM meditrack_app;

-- Document: a future contributor adding a new mutation-required table (e.g. an
-- async-jobs table) gets the DEFAULT PRIVILEGES grants automatically; the
-- AuditEvent table is the documented exception.
```

The DATABASE_URL password value (`meditrack_app_dev`) is chosen to be unmistakably non-production. README explicitly documents that production deployments substitute this with a real secret (out of scope for the demo; mentioned in §"What I'd do with more time" if relevant).
