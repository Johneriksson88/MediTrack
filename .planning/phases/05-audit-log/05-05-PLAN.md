<objective>
Close three smaller VERIFICATION.md gaps that share a common shape — each is a one-or-two-file backend hygiene fix with a focused regression test — but each independently affects a graded surface (error taxonomy, forensic attribution, filter API integrity).

The three gaps are bundled into one plan because they are all backend-only, all use existing primitives (no new dependencies, no new patterns), and each carries a single regression test against the same `audit.integration.test.ts` suite. Splitting them into three separate plans would inflate the wave count without semantic benefit.

Purpose:
- **CR-02 (BLOCKER):** `audit.service.ts:85-87`'s `decodeCursor` throws `ValidationFailedError` with `details.reason: 'invalid_quantity'` — the order-line validation reason code. Cross-subsystem taxonomy leak. The fix is one literal swap + a regression test asserting the response's `details.reason` is NOT `'invalid_quantity'` and IS `'invalid_cursor'`. Without this, any FE switching on `details.reason` to localize error toasts mis-localizes a cursor failure as a quantity failure.
- **CR-04 (BLOCKER):** `auth.ts:51-65`'s `DELETE /api/auth/session` handler has no `setActor()` call before `logout(unsigned.value)`. The ALS store's `actorUserId` is null when the `$extends` middleware writes the `auth.logout` row, so every production logout audit row records `actorUserId: null`. "Who logged out user X" is unanswerable — the entire point of an audit log. The fix is `findSessionById(unsigned.value)` → `setActor(session.userId, session.careUnitId, req.ip)` → `logout(unsigned.value)`. The route stays unprotected (idempotent logout per Phase 1 D-01 — no `requireSession` preHandler added).
- **WR-07 (PARTIAL):** `auth.service.ts:64`'s `entityId: ''` sentinel for the unknown-email failed-login row is schema-permitted (`entityId String NOT NULL` admits empty strings) but semantically wrong. The admin filter `?entityType=session&entityId=` would match all unknown-email failed-login rows AND any other row with missing entityId. Fix: extend `AUDIT_ENTITY_TYPES` with `'auth_attempt'`, use the attempted email as `entityId` for the unknown-email branch (structurally meaningful — an admin investigating brute-force can filter by attempted-email-as-entityId). The known-user-wrong-password branch is unchanged (it already records `entityType: 'session'`, `entityId: user.id`).

Output: Three tiny backend changes, one shared-constant extension, three new regression tests.
</objective>

<why_this_approach>

**CR-02 — pick `'invalid_cursor'`.** D-19 (the canonical error envelope, locked Phase 1 decision) sets the `details` shape as `unknown` — any keys are admissible. Adding a new `details.reason` literal `'invalid_cursor'` is NOT a new error CODE (top-level `code` stays `validation_failed`), it is a new `details.reason` value. CONTEXT.md D-19 explicitly notes "introduces NO new error codes"; that constraint is honored. The literal `'invalid_cursor'` is grep-discoverable, matches the natural domain noun, and avoids the cross-subsystem leak that `'invalid_quantity'` causes.

**CR-04 — lookup-before-destroy, route stays unprotected.** The verification report names this approach explicitly. Why NOT add `requireSession` as a preHandler:
1. Phase 1 D-01 locks logout as idempotent — a client without a valid cookie still receives 204. `requireSession` would return 401, breaking the contract.
2. The fix is "resolve the actor manually before destroying the session", which is a 3-line change in the handler. We use `findSessionById` (which already exists in `apps/api/src/auth/session.ts:41-43`).
3. If the cookie is missing OR the session doesn't exist OR the unsign fails, the `setActor` call is simply skipped — the logout still returns 204 (idempotency preserved), the audit row simply isn't written (no Session row to delete; the `$extends` middleware doesn't fire). This matches the existing happy path semantics.

**WR-07 — option (a), extend AUDIT_ENTITY_TYPES with 'auth_attempt'.** Verification report's recommendation. Reasons:
1. `'auth_attempt'` is a structurally meaningful entity type — the entity IS "an authentication attempt", which is distinct from a Session (the persistent post-login object). Conflating both under `'session'` is the cause of WR-07.
2. The attempted email becomes the entityId — a forensically useful identifier (an admin investigating brute-force can filter `?entityType=auth_attempt&entityId=alice@example.com` to see every failed attempt for that email, regardless of whether the email maps to a real User).
3. AUDIT_ENTITY_TYPES is already a string-union open-on-the-API surface (per the constant's docstring lines 13-15: "The set is OPEN on the API boundary"); adding `'auth_attempt'` requires (a) the const-list append, (b) the Swedish label addition in AUDIT_ENTITY_TYPE_LABELS, (c) NO schema migration (entityType is `String`, not a Postgres enum, per CONTEXT.md "Claude's Discretion" line 106). Three locations, zero downstream-FE changes (the FE label map auto-broadens via type inference).
4. The known-user-wrong-password branch (auth.service.ts:79-99) STAYS unchanged: `entityType: 'session'`, `entityId: user.id`. The user is real; the session is the conceptually-attempted-but-unrealized object; the existing convention holds. Only the unknown-email branch (auth.service.ts:48-74) changes — that branch is the one where no User exists and no Session would have been created.

Why bundle CR-02 + CR-04 + WR-07 into ONE plan:
- All three are backend-only file edits with focused regression tests.
- No file conflicts (CR-02 touches audit.service.ts only; CR-04 touches auth.ts + has incidental imports; WR-07 touches auth.service.ts + shared constants).
- All three regression tests land in the same `audit.integration.test.ts` file (which Task 4 modifies in Plan 04 — but Plan 05 lands in a different test block and the merge is trivial).
- Splitting into three plans would not change the wave count (all three are wave 4); it would just inflate the SUMMARY file count and dilute the commit narrative.

**No CONTEXT.md amendment needed.** CR-02 uses an existing-shape error envelope. CR-04 implements D-92 verbatim (which the original Plan 01 missed). WR-07 extends an open-shape const-list per its own docstring's documented extension path.

</why_this_approach>
