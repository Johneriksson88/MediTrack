<why_this_approach>
Two viable options from VERIFICATION.md `gaps:` block:

  (a) `Prisma.getExtensionContext(this)` inside each handler — resolve the active client (which is the tx when the caller is inside `prisma.$transaction(async tx => ...)`) and route BOTH the pre-load reads AND the audit-row INSERT through it.

  (b) A service-layer `withAudit(tx, ...)` wrapper called explicitly inside every `prisma.$transaction` block.

Option (c) — "accept the limitation, drop the D-91 claim from README" — is OFF THE TABLE; D-91 is a locked CONTEXT.md decision.

**We pick option (a).** Reasons:

1. D-83 / D-90 promise "retrofit without touching prior-phase service code". Option (b) requires editing every `prisma.$transaction(async tx => ...)` block in `order.service.ts` (and any future service that opens a tx) to call `await withAudit(tx, ...)` — this directly violates D-83.

2. `Prisma.getExtensionContext(this)` is the documented Prisma API for extensions that need access to the active client surface. Prisma's own docs describe this as the canonical way to write extensions that work inside transactions. We're not inventing a new pattern; we're using the one that exists for this exact problem.

3. The handler's `this` is bound by Prisma to the active surface — inside `tx.order.updateMany(...)` from within `prisma.$transaction(async tx => ...)`, `this` resolves to the tx client. The `getExtensionContext` helper unwraps it to the same surface our handlers need for `findUnique` / `findMany` / `auditEvent.create`.

4. Service code (`order.service.ts`, `auth.service.ts`, future services) does not change. The fix is local to `auditExtension.ts`.

This is the same general approach as Prisma's own [audit log middleware example](https://www.prisma.io/docs/orm/prisma-client/client-extensions/query#modify-the-arguments-of-a-prisma-client-operation) — although we additionally need it INSIDE arbitrary user-opened transactions, which is exactly what `getExtensionContext` is for.

**Note on the README claim:** "If the mutation rolls back, the audit row rolls back with it" becomes TRUE again once the fix lands. Sequence: ship the auditExtension refactor + regression test FIRST in this plan's task ordering, README update LAST — so the README is never in a state where it disagrees with HEAD.

**No CONTEXT.md amendment is needed.** D-91 was always the locked contract; this plan implements it correctly.
</why_this_approach>
