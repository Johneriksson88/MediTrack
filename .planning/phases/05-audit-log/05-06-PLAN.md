<objective>
Restructure the ALS layer underlying the audit pipeline so the entire class of "shared mutable slot under concurrency" bugs is structurally eliminated, not patched. Closes CR-01 (nested + parallel tx attribution) and CR-04 (cross-request keep-alive leakage) from 05-REVIEW.md AND addresses the high-priority concerns in 05-REVIEWS.md (HIGH #1 per-concern ALS, HIGH #2 activeTxStack-as-default).

The original Plan 06 design (save/restore on a single shared store) was the local-optimization fix. The cross-AI review (05-REVIEWS.md) flagged that design as still fragile: a fourth slot added carelessly in v2 would re-introduce the same class of bug. The structural fix — three independent `AsyncLocalStorage` instances, each used with `.run(value, fn)` — eliminates the bug class entirely by relying on Node.js stdlib's documented save-restore-via-stack-frames semantics. The activeTxStack-as-default decision (HIGH #2) is a strict-subset upgrade of the original Plan 06 stack fallback — shipping the stack from day 1 is safer with negligible runtime cost.

Purpose:

- **HIGH #1 (05-REVIEWS.md) — Per-concern ALS instances.** Replace the single `RequestContext` store (with fields `{actorUserId, careUnitId, requestId, requestSource, ipAddress, actionOverride, activeTx}`) with three independent AsyncLocalStorage instances: `actorALS` (carries the request scope: actor + careUnit + requestId + requestSource + ipAddress), `activeTxStackALS` (carries a stack of tx clients), `actionOverrideALS` (carries the temporary domain-rich action). Each storage is used exclusively via `.run(value, fn)` — no `.enterWith`, no manual mutation of the store object. The shared-mutable-slot anti-pattern is gone; the entire CR-01 class of bugs (asymmetric clear, nested mis-restore, parallel race) cannot recur because each ALS instance's frame lifetime is bounded by the closure passed to `.run`.

- **HIGH #2 (05-REVIEWS.md) — activeTxStack ships as the default.** The original Plan 06 design shipped save/restore on a single slot with a stack fallback "if the parallel test fails." The review report (05-REVIEWS.md) recommends shipping the stack from day 1. This plan does so. `activeTxStackALS` carries an array; `patchTransactionForAudit` pushes the tx into the array on entry to the interceptor (via `activeTxStackALS.run([...prev, tx], () => ...)` for proper save/restore) and the per-model handler reads `activeTxStackALS.getStore()?.at(-1)` for the top of the stack. Nested calls get a stack with two entries; the outer's tail mutation after the inner returns reads the outer tx; parallel calls each have their own ALS frame from the start (no race because each `.run` callback has its own immutable frame snapshot). Implementation detail: because `.run(value, fn)` is the only set-mode for ALS, the "stack" is actually a new array allocated per call (containing the previous stack's entries + the new tx); this is the cleanest fit with ALS semantics and costs O(depth) allocation per tx — negligible.

- **CR-04 (05-REVIEW.md) — Fastify 3-arg onRequest + actorALS.run.** Same fix shape as the original Plan 06 design, but applied to actorALS instead of the merged store. The plugin's hook becomes `app.addHook('onRequest', (req, reply, done) => actorALS.run({actorUserId: null, careUnitId: null, requestId, requestSource: 'http', ipAddress}, () => done()))`. The `done()` continuation advances the request pipeline inside the ALS frame.

- **Test hardening (LOW #13 + #16 from 05-REVIEWS.md).** The parallel-tx regression test asserts deterministic interleaving via `setImmediate` between the two `prisma.$transaction` callbacks, and uses a HARD `expect(cums.length).toBeGreaterThanOrEqual(2)` instead of a silent `console.warn` skip. The seed guarantees ≥2 CUM rows; if a future seed regression drops below 2 the test fails loudly.

- **README documentation.** §How the audit hook works updated to describe the per-concern ALS design verbatim. A new §"Lessons learned" subsection (or expanded §"What I'm proud of") captures: (a) the refactor away from shared-mutable-slot as the right closure for CR-01/CR-04; (b) the Node.js docs explicitly warning against `als.enterWith` for request-context use (link to https://nodejs.org/api/async_context.html — "Use of asyncLocalStorage.enterWith() in production code is generally discouraged") — this is the "consult the stdlib docs before introducing a primitive" process retro.

<why_this_approach>

**Per-concern ALS instances (HIGH #1).** The original RequestContext shape collapsed three independent concerns onto one mutable object:
- Actor + careUnit + requestId + ipAddress (request-scope identity — set once at request entry, read many times throughout the request)
- activeTx (mutation-scope — set on $transaction entry, read on per-model handler invocation, cleared on $transaction exit)
- actionOverride (call-scope — set on withActionOverride entry, read on handler invocation, cleared on callback return)

These three concerns have three different lifetimes. Cramming them onto one store and mutating fields in-place was the source of every bug 05-REVIEW.md found:
- CR-01 nested case: inner $transaction's finally cleared activeTx to undefined; the outer's actor/requestId fields stayed correct, but the slot the outer needed at tail time was gone.
- CR-01 parallel case: same root cause — both interceptors race on the same slot regardless of save/restore on a single-slot design.
- CR-04 keep-alive case: enterWith bound the store to whichever async context happened to be live when the request entered; the store object survives request boundaries because there's only one of it.
- WR-XX class (Plan 05 found three of these): asymmetric clear, omitted set, hidden interaction between slots.

The fix is to give each concern its own AsyncLocalStorage instance with no shared mutable fields. `.run(value, fn)` is the ONLY way to set a value; on the way out, Node's stdlib restores the previous frame from the call stack automatically. Concurrent `.run` calls get independent stack frames; parallel `Promise.all` of two `.run` callbacks get independent frames; subsequent requests get fresh frames. The fragility class is structurally gone.

Why NOT keep `RequestContext` as a single store with save/restore on activeTx (the original Plan 06 design): the save/restore alone is the LOCAL fix; per-concern ALS is the STRUCTURAL fix. The cost difference is ~30 LOC; the benefit difference is the entire class of future bugs. 05-REVIEWS.md HIGH #1 is exactly this trade-off, and the review's recommendation is unambiguous: ship the structural fix.

Why NOT use a single `AsyncLocalStorage<RequestContext>` with `.run` (instead of `.enterWith`) + save-restore inside the existing helpers: still leaves the shared-mutable-slot store object. Same fragility for v2 fields.

**activeTxStack-as-default (HIGH #2).** Two arguments:
1. **Correctness:** under Prisma's interactive-tx model, every `$transaction(callback)` invocation creates an independent client (`tx`). The audit interceptor wraps that callback in `activeTxStackALS.run([...prev, tx], fn)`. Inside `fn`, every Prisma model operation on `tx` triggers a per-model handler; the handler reads `activeTxStackALS.getStore()?.at(-1)` and gets `tx` (top of stack). Nested calls give a stack `[outerTx, innerTx]`; the inner handler reads `innerTx`; on inner exit the stack reverts to `[outerTx]`; the outer's tail mutations read `outerTx` correctly. Parallel calls each have their own initial frame, so there's no cross-attribution.
2. **Cost:** allocating a new array on each push is O(stack depth) — typically 1 entry, sometimes 2 for nested calls. Negligible vs the Prisma round-trip latency the audit row INSERTs already pay.

Why NOT use a mutable array on a single ALS instance (push/pop in place): would re-introduce the mutation-across-frames hazard. The whole point is to use `.run`'s immutable-frame semantics.

Why NOT use `Symbol.dispose` / `using` statements for tx lifetime (TS 5.2+): Node 20 LTS doesn't support `using` semantics natively at runtime yet; the polyfill complicates the codebase for no architectural benefit over ALS.

**CR-04 als.run on the 3-arg Fastify onRequest hook.** Same rationale as the original Plan 06 design — Fastify's `done()` continuation is the documented way to advance the request pipeline; wrapping `done()` inside `actorALS.run(scope, () => done())` makes the rest of the pipeline run inside the ALS frame. Under sequential request load this is observably identical to als.enterWith; under concurrent keep-alive load (Test 14 in the new suite), als.run guarantees per-request isolation that als.enterWith only approximated.

Why NOT fall back to als.enterWith if als.run breaks Fastify's preHandler/handler/onResponse chain (the original Plan 06 escape hatch): the previous plan documented this as "if Fastify's done() doesn't preserve the ALS frame, revert." Empirical evidence from Fastify's own docs (https://fastify.dev/docs/latest/Reference/Hooks/) confirms `done()` propagates through the hook chain; the hook system is designed for this pattern. The escape hatch is removed from this plan — if Test 14 fails, the executor investigates the failure directly rather than reverting.

**Bundling all three fixes into one plan.** They share a single architectural concern (ALS-store-as-shared-mutable-slot leaks across concurrency boundaries) and a single root-cause refactor (per-concern ALS instances). Splitting CR-01 vs CR-04 would create two adjacent commits modifying the same three files (`requestContext.ts`, `auditExtension.ts`, `audit.integration.test.ts`) with overlapping diffs — documentation noise without semantic benefit. The plan ships as a single atomic refactor.

**No CONTEXT.md amendment needed.** D-91 (same-tx audit contract) and D-92 (ALS actor attribution) are the locked contracts. This plan implements them correctly under concurrency; the decisions don't change. D-94 (actionOverride scoping) is preserved verbatim — `actionOverrideALS` carries the override, `withActionOverride` calls `actionOverrideALS.run(action, fn)`.

```typescript
// Pre-plan: stored tx into store.activeTx, cleared in finally.
// Post-plan: wraps original transaction in withActiveTx(tx, fn).
export function patchTransactionForAudit<T extends { $transaction: (...args: any[]) => Promise<any> }>(extendedClient: T): T {
  const original$transaction = (extendedClient as any).$transaction.bind(extendedClient);
  Object.defineProperty(extendedClient, '$transaction', {
    value: async function (fnOrOps: unknown, options?: unknown): Promise<unknown> {
      if (typeof fnOrOps !== 'function') {
        return original$transaction(fnOrOps, options); // batch-of-promises form, no audit interception
      }
      const actor = actorALS.getStore();
      if (!actor) {
        return original$transaction(fnOrOps, options); // no ALS frame → no audit; seed-suppression
      }
      return original$transaction(async (tx: any) => {
        return withActiveTx(tx, () => (fnOrOps as (tx: any) => Promise<unknown>)(tx));
      }, options);
    },
    writable: true,
    configurable: true,
  });
  return extendedClient;
}
```
