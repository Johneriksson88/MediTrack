Close two CRITICAL findings from 05-REVIEW.md — both materially threaten the D-91 / D-92 audit-row attribution contract that Plan 04 nominally closed but only verified under serial, single-tx execution. The verifier flagged these as latent contract violations that the existing regression test suite cannot falsify.

The two findings are bundled into one plan because they share a single root cause (the shared-mutable-slot design of the ALS store) and are mitigated by the same symmetric save/restore + scope-bounding discipline.

Purpose:

- **CR-01 (BLOCKER, 05-REVIEW.md:54-109):** `apps/api/src/db/auditExtension.ts:344-385`'s `patchTransactionForAudit` interceptor sets `store.activeTx = tx` before the user callback and clears it with `store.activeTx = undefined` in `finally`. This asymmetric clear breaks two scenarios that single-threaded happy-path tests cannot exercise: (a) **nested `$transaction`** — outer opens tx, inner opens tx, inner finally clears the slot to undefined, the outer's tail mutations after the inner-block now resolve `activeClient = store.activeTx ?? client` to `client` (the root pool), re-introducing the original orphan-audit-row class of bug Plan 04 was supposed to close; (b) **parallel `Promise.all([prisma.$transaction(fnA), prisma.$transaction(fnB)])`** — both interceptor calls race on the same slot, the second push clobbers the first, then either finally pulls the rug out from audit writes still in flight in the other branch. Fix: mirror the `withActionOverride` save/restore pattern verbatim — `const previous = store.activeTx; store.activeTx = tx; try {...} finally { store.activeTx = previous; }`. Two regression tests prove the fix: (1) nested tx with outer-tail-rollback after inner-success leaves zero audit rows for BOTH the outer- and inner-tx mutations; (2) parallel tx writes one audit row per tx with no cross-attribution.

- **CR-04 (BLOCKER, 05-REVIEW.md:216-269):** `apps/api/src/plugins/requestContext.ts:128-160`'s `requestContextPlugin` calls `als.enterWith({...})` inside its onRequest hook. Under Fastify keep-alive, an async chain that has not yielded back to the event loop in a way that creates a fresh async resource may observe a subsequent request's store mid-tail — combined with the activeTx slot from CR-01 (and the actorUserId, careUnitId slots), this routes audit-row INSERTs through the WRONG tx client and attributes them to the WRONG user. The prior 05-REVIEW.md flagged this as WR-01 carrying lower severity; the introduction of the activeTx slot in Plan 04 made the blast radius materially worse ("the audit row may commit or roll back with a different mutation than the one it claims to describe"). Fix (option 1 from review, preferred): convert the onRequest hook to the Fastify 3-arg `(req, reply, done) => als.run(scope, () => done())` signature. The `done()` continuation is Fastify's documented way to advance the request pipeline; wrapping it inside `als.run` keeps the rest of the request lifecycle inside the ALS frame. A cross-request smoke test fires three concurrent requests through one keep-alive `buildTestApp()` instance and asserts each audit row's actorUserId matches the originating user — no actorUserId↔requestId cross-attribution.

Both fixes are local: CR-01 is a single-function edit in auditExtension.ts; CR-04 is a single-hook edit in requestContext.ts. No service files change. No new packages. No migration. <why_this_approach>

**CR-01 — symmetric save/restore mirroring withActionOverride verbatim.** The review report names this fix shape explicitly and the codebase already has the analog: `requestContext.ts:111-126`'s `withActionOverride` is exactly this pattern (save previous, set new, try{} finally{ restore previous }). Mirroring it for `activeTx` makes the two helpers structurally identical and makes future contributors who grep `const previous = store.` find both call sites together. The asymmetric `= undefined` clear was the documented Plan 04 design — its bug is the conflation of "release the slot so it doesn't leak into the next bare call" (the intent) with "always reset to undefined" (the implementation). Save/restore preserves both goals: the slot returns to whatever it was before THIS interceptor invocation (which is `undefined` at the outermost layer and `tx_outer` at a nested layer), and concurrent calls each see their own restored value because each interceptor invocation has its own `previous` closure variable.

Why NOT take the alternative "make activeTx a stack" approach (push on entry, pop on exit): a stack would also work, but it's more code and a new design pattern in the codebase. The codebase already has save/restore; using it twice is the lighter footprint. The stack-based alternative is documented in the review report as a fallback but not the preferred fix.

The parallel-tx case requires a more subtle argument: even with save/restore, two parallel `prisma.$transaction` calls share the SAME ALS store object (the store is created once per request by the onRequest hook). When tx-A's interceptor sets `store.activeTx = txA` and tx-B's interceptor immediately after sets `store.activeTx = txB`, tx-A's per-model handlers now read `txB` from the slot — cross-attribution. **However**, this works ONLY if the two callbacks run interleaved on the event loop with their handlers reading the slot between A's assignment and B's assignment. Under Prisma's interactive-tx model, each `$transaction(async (tx) => {...})` callback completes synchronously (within the async sense) before the audit handlers fire — the handlers are intercepted ON THE tx client, not on the root. So the handler invocations are `txA.<model>.<op>` (which routes through the tx-A interceptor chain) and `txB.<model>.<op>` (tx-B's chain). The activeTx slot is read inside the handler; if both handlers run concurrently, both could read whichever value happens to be in the slot. Save/restore alone does not fix the race. **But** the per-model handler logic resolves `activeClient = store.activeTx ?? client`, and Prisma's `getExtensionContext(this)` (called inside each handler in Plan 04's failed attempt) actually does return the tx client when the handler is invoked from inside a tx context. Plan 04 SUMMARY documented that `getExtensionContext` is a no-op identity function in Prisma's runtime, hence the activeTx slot was introduced. The correct fix for the parallel case is therefore TWO-PART: (1) save/restore so the nested case works; (2) the parallel-test exercise reveals whether the activeTx slot is sufficient under concurrency, OR whether a stack is needed. The plan's task ordering lets the parallel-test (Task 2's parallel-tx case) reveal the truth empirically — if save/restore alone passes the parallel regression test (because Prisma's interactive-tx callback model serializes the handlers within one tx — each tx's handlers run sequentially even if multiple txs run concurrently), then save/restore is sufficient. If the parallel test fails, the executor implements the stack fallback (documented in the task action) before commit.

**CR-04 — als.run(scope, () => done()) using Fastify's 3-arg onRequest hook signature.** Review report option 1. Reasons:
1. Fastify documents this signature: `app.addHook('onRequest', (req, reply, done) => { ... done(); })` — calling `done()` advances the request pipeline (preHandler → handler → onSend → onResponse). Wrapping `done()` inside `als.run(scope, () => done())` makes the rest of the pipeline run inside the ALS frame.
2. Node.js `AsyncLocalStorage.run(store, fn)` documentation guarantees that `fn` and everything it transitively awaits runs inside the store's frame. Fastify's continuation pattern composes naturally with this — `done()` schedules the next hook in the chain, but the await chain rooted at the `als.run` callback keeps the frame alive until the whole request completes.
3. The conversion is structurally minimal: the existing async-function form `app.addHook('onRequest', async (req, reply) => { als.enterWith(scope); })` becomes the 3-arg sync form `app.addHook('onRequest', (req, reply, done) => { als.run(scope, () => done()); })`. The Fastify type signature for the 3-arg form is `(req, reply, done: HookHandlerDoneFunction) => void`. No new dependencies; no new code paths.
4. Behavior preservation: under sequential request load (the existing test suite), the new signature is observably identical to the old — every audit row continues to be attributed to the correct request. The change only matters under concurrent / keep-alive load, which is exactly the case the CR-04 cross-request smoke test exercises.

Why NOT option 2 (document as known gap + ship a skipped failing test): option 1 is non-trivial but well-trodden — Fastify ships first-class support for this exact pattern, and the conversion is one hook. Option 2 was the fallback the review report named in case option 1 turned out to be infeasible. The plan's task ordering surfaces infeasibility empirically: Task 4 (CR-04 als.run conversion) must keep the existing audit integration suite passing as a precondition for shipping. If the existing suite breaks (because Fastify's `done()` continuation does NOT preserve the ALS frame as expected), the executor falls back to option 2 with documented evidence — but does not punt without trying option 1 first.

**Bundling CR-01 + CR-04 into one plan.** Reasons:
- Both close the same architectural concern: ALS-store-as-shared-mutable-slot leaks across concurrency boundaries. CR-01 fixes the within-request leak (nested + parallel txs); CR-04 fixes the across-request leak (keep-alive Fastify connections). They are two sides of the same coin.

export function patchTransactionForAudit<
  T extends { $transaction: (...args: any[]) => Promise<any> },
>(extendedClient: T): T {
  const original$transaction = (extendedClient as any).$transaction.bind(extendedClient);

  Object.defineProperty(extendedClient, '$transaction', {
    value: async function (fnOrOps: unknown, options?: unknown): Promise<unknown> {
      if (typeof fnOrOps !== 'function') {
        return original$transaction(fnOrOps, options);
      }
      const store = als.getStore();

        return original$transaction(fnOrOps, options);

          store.activeTx = undefined; // <-- CR-01: this asymmetric clear breaks nested + parallel cases
        }
      }, options);
    },
    writable: true,
    configurable: true,
  });

  return extendedClient;
}
