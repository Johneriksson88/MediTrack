<why_this_approach>

**`@fastify/rate-limit` over a custom in-process counter (MEDIUM #8).** The plugin is the documented Fastify ecosystem solution; it ships:
- Per-route configurable limits via the route's `config.rateLimit` block.
- Customizable keyGenerator (we use `email + '|' + ip`).
- Built-in 429 response with `Retry-After` header.
- In-memory store by default (fits the single-process docker-compose demo); Redis store available for multi-process production.
- TypeScript types out of the box.

A custom counter (a Fastify preHandler maintaining a Map of email â†’ attempt timestamps) would be ~30 lines of code with edge cases (timestamp cleanup, memory leak prevention, race conditions); the plugin handles all of that.

Why NOT skip the rate-limit and just rely on a future Phase 7 ops layer (nginx, Cloudflare): the brief asks for an answer to "what would you do to harden auth," and the README's Â§6 narrative is strengthened by having a concrete rate-limit in the demo code. A note in README that "production would also add CDN-layer rate-limiting" is the v2 mention.

**Per-email + per-IP combined keyGenerator.** Two attack patterns:
- (a) Brute-force one email from many IPs (botnet credential stuffing). Per-IP bucket alone doesn't catch this; per-email bucket does.
- (b) Iterate through emails from one IP (slow scan). Per-email bucket alone doesn't catch this; per-IP bucket does.

The keyGenerator `email + '|' + ip` creates a per-(email,IP) bucket. To also bound per-email globally and per-IP globally, we register the plugin TWICE â€” once per route with the combined key, once globally with a per-IP key. The combined key has max 10 / minute; the per-IP key has max 30 / minute.

Alternative considered: a Redis-backed shared counter for production. Out of scope for v1 â€” README mentions it.

**Rate-limit-rejected requests do NOT write an audit row.** Rationale:
- An auth.login_failed audit row documents a REAL login attempt (verifyCredentials ran, decided fail). A rate-limit rejection is the absence of an attempt â€” verifyCredentials never ran.
- Writing audit rows for rate-limit rejections would defeat the rate-limit's row-growth-bounding purpose.
- The rate-limit rejection IS visible â€” it produces a 429 response, which is logged by Fastify's request logger; if forensic visibility of rate-limit hits is needed in v2, that's a separate audit category ('auth.rate_limited' for example), not the existing auth.login_failed.

**Inline INVARIANT comment + integration test (MEDIUM #7 + LOW #19).** The two work together: the comment documents the assumption AT THE SITE; the test ENFORCES the assumption at CI time. Either alone is weaker â€” comments rot without tests; tests without commented context confuse future readers.

The test wraps a synthetic flow: open a `prisma.$transaction`, call a function that explicitly writes an auth.login_failed audit row (NOT using the prisma client passed to the tx callback â€” using the outer prisma singleton), throw to force rollback, assert the row is still present after. If a future contributor accidentally pipes the tx client into the audit write (e.g., refactoring to use the per-extension prisma object that runs inside the tx), the test fails.

Why NOT skip the test and rely on the comment alone: comments do not survive refactors. The test is the binding contract.

**Bundling.** The three findings concern the same code path (POST /api/auth/login + verifyCredentials + the auth.login_failed write). Bundling keeps the auth-side hardening atomic; splitting would create overlapping diffs in auth.ts and auth.service.ts across two commits.

</why_this_approach>
