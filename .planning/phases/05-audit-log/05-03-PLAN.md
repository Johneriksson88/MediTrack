**Step A — extend `buildTestApp.ts`** if needed: add `export async function resetAuditEvents(): Promise<void> { await prisma.$executeRawUnsafe('TRUNCATE "AuditEvent" CASCADE;'); }`. Note: `$executeRawUnsafe` bypasses the audit extension (extension intercepts model methods, not raw SQL) AND it requires DELETE/TRUNCATE privilege — but `TRUNCATE` is what was REVOKEd. The test runner will need a separate DB role with sufficient privilege OR use `prisma.auditEvent.deleteMany({})` — wait, that's also REVOKEd at the DB layer AND banned by ESLint. **Solution:** the test setup runs as the same Postgres role that has the REVOKE applied; TRUNCATE will FAIL with permission denied (which is exactly what test #3 asserts). For test cleanup, use a privileged role. Two paths — pick one:
  - Option A: configure `DATABASE_URL` for tests to a separate superuser/owner role that retains UPDATE/DELETE/TRUNCATE; the application's REVOKE applies only to the app role. This is the more realistic production posture (the migration runner / table owner has full grants; the app's connection role does not).
  - Option B: in test setup, use raw DDL `ALTER TABLE "AuditEvent" DISABLE TRIGGER ALL` / re-GRANT and re-REVOKE around the TRUNCATE.

**Take Option A** — modify the REVOKE migration in Plan 01 if it isn't already structured this way: the migration revokes from the application's connection role (the role embedded in DATABASE_URL when the application runs), NOT from the migration runner role. In dev/test, both are often the same Postgres role; to keep test cleanup working, document in the test helper: `resetAuditEvents()` SHOULD work because the test connection role IS the migration runner role (Phase 1's DATABASE_URL points at a role with full DDL grants). **If the test fails with permission denied on TRUNCATE, that itself is a positive signal of the REVOKE working — but it breaks test isolation.** Resolution: in `buildTestApp.ts`'s `resetAuditEvents` helper, use `prisma.$executeRawUnsafe('DELETE FROM "AuditEvent"')` and document that this runs as the test/migration role which still has DELETE grants on the table (the REVOKE only takes effect for the application's runtime role; the test role is the table OWNER and retains all grants). This needs to be verified against the actual REVOKE migration syntax — if the migration says `REVOKE ... FROM CURRENT_USER` and `CURRENT_USER` at migration time IS the test role, then test cleanup is impossible.

**Final decision:** the REVOKE migration's `CURRENT_USER` resolves to the connection role at migration time, which in dev/test is typically the same role. To make the tests pass:
  1. **In Plan 01 Task 1**, the REVOKE migration must use an explicit role name, NOT `CURRENT_USER` — e.g. read it from an env var like `MEDITRACK_APP_ROLE` defaulting to the role name in `DATABASE_URL`. **Update the Plan 01 acceptance criteria post-hoc here:** if the executor in Plan 01 Task 1 used `CURRENT_USER` literally, Plan 03 Task 2 must amend the REVOKE migration to use an explicit role.
  2. **OR** — simpler — the test helper grants UPDATE/DELETE/TRUNCATE back to itself temporarily: `prisma.$executeRawUnsafe('GRANT UPDATE, DELETE, TRUNCATE ON "AuditEvent" TO CURRENT_USER')` before TRUNCATE, then re-REVOKE after. This violates the spirit of the test (re-granting defeats the layer) — DO NOT do this.

**Pragmatic resolution that doesn't require amending Plan 01:** The test #3 (DB-layer REVOKE assertion) runs `$executeRawUnsafe` and expects `permission denied`. The other 4 tests don't need to TRUNCATE the audit table — they can either (a) accept incrementally-growing rows and use unique entityId markers to filter, OR (b) the test harness creates a SEPARATE Postgres role for the app connection that has REVOKE applied while the migration runner / test cleanup uses a different role. **Recommendation: skip cleanup of AuditEvent between tests** — each test asserts properties of rows it just created (filtered by entityType + entityId + timestamp window) without depending on the table being empty. This is the cleanest path and respects the spirit of "audit log is append-only."

import { afterAll, beforeAll, beforeEach, describe, expect, it } from 'vitest';
import { execFileSync } from 'node:child_process';
import type { FastifyInstance } from 'fastify';
import {

import { als } from '../src/plugins/requestContext.js';

let app: FastifyInstance;

beforeAll(async () => {
  app = await buildTestApp();
  await ensureAllRolesSeeded();

beforeEach(async () => {
  await resetSessions();
});
afterAll(async () => {
  await app.close();
  await prisma.$disconnect();

describe('AUD-01 — full pipeline coverage', () => {
  it('writes audit rows for every step of create-draft → submit → confirm → deliver', async () => {
    // Mark this test run with a timestamp so we can filter audit rows added during this test
    const testStartedAt = new Date();
    // Login as nurse
    const nurseLogin = await loginAs(app, TEST_SJUKSKOTERSKA.email, TEST_SJUKSKOTERSKA.password);
    const nurseCookie = captureSessionCookie(nurseLogin.headers['set-cookie']);
    // Login as apotekare
    const apotekareLogin = await loginAs(app, TEST_APOTEKARE.email, TEST_APOTEKARE.password);
    const apotekareCookie = captureSessionCookie(apotekareLogin.headers['set-cookie']);
    // Create order + 2 lines + submit + confirm
    const orderId = await createEmptyOrder(app, nurseCookie);
    const cum1 = await findTestCareUnitMedication(TEST_SJUKSKOTERSKA.careUnitId);
    // (one CUM is enough — D-94's 1+N still works with N=1)
    await progressOrderToBekraftad(nurseCookie, apotekareCookie, orderId, [{ cumId: cum1.id, quantity: 5 }]);
    // Deliver
    const deliverRes = await app.inject({
      method: 'POST', url: `/api/orders/${orderId}/deliver`,
      cookies: { 'meditrack.sid': apotekareCookie },
    });
    expect(deliverRes.statusCode).toBe(200);

    // Now: query AuditEvent for rows created since testStartedAt for this order id
    const rows = await prisma.auditEvent.findMany({
      where: { createdAt: { gte: testStartedAt } },
      orderBy: { createdAt: 'asc' },
    });
    // Expected actions in order:
    // 1. auth.login (nurse)
    // 2. auth.login (apotekare)
    // 3. create (order)
    // 4. create (orderLine x N)
    // 5. order.submit (order updated by nurse)
    // 6. order.confirm (order updated by apotekare)
    // 7. order.deliver (order updated by apotekare)
    // 8. stock.increment (CUM x N, sharing requestId with order.deliver)
    expect(rows.find(r => r.action === 'auth.login' && r.actorUserId === apotekareLogin.body /* user.id */)).toBeDefined();
    expect(rows.find(r => r.action === 'order.submit' && r.entityId === orderId)).toBeDefined();
    expect(rows.find(r => r.action === 'order.confirm' && r.entityId === orderId)).toBeDefined();
    const deliverRow = rows.find(r => r.action === 'order.deliver' && r.entityId === orderId);
    expect(deliverRow).toBeDefined();
    const stockIncRows = rows.filter(r => r.action === 'stock.increment' && r.requestId === deliverRow!.requestId);
    expect(stockIncRows.length).toBeGreaterThanOrEqual(1);
    // 1+N sibling shape (D-94): order.deliver + stock.increment share requestId
    expect(stockIncRows.every(r => r.requestId === deliverRow!.requestId)).toBe(true);
  });

  it('rolled-back mutations leave zero audit rows (D-91)', async () => {
    // Use Phase 4's submit-failure trigger: submit an order with zero lines → 422 + rolled-back tx
    const testStartedAt = new Date();
    const nurseLogin = await loginAs(app, TEST_SJUKSKOTERSKA.email, TEST_SJUKSKOTERSKA.password);
    const nurseCookie = captureSessionCookie(nurseLogin.headers['set-cookie']);
    const orderId = await createEmptyOrder(app, nurseCookie);
    // POST submit with no lines → 422 validation_failed (Phase 3 contract)
    const submitRes = await app.inject({
      method: 'POST', url: `/api/orders/${orderId}/submit`,
      cookies: { 'meditrack.sid': nurseCookie },
    });
    expect(submitRes.statusCode).toBe(422);
    // No order.submit audit row should exist for this orderId
    const submitAuditRows = await prisma.auditEvent.findMany({
      where: { action: 'order.submit', entityId: orderId, createdAt: { gte: testStartedAt } },
    });
    expect(submitAuditRows).toHaveLength(0);
  });
});
