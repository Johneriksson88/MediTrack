Entity-type set: `medication`, `care_unit_medication`, `order`, `order_line`, `user`, `session`.

EntityId resolution rule (D-97 + T-05-03):
- `Session` → audit row's `entityId = row.userId` (the auditable owner). NEVER `row.id` — that is the raw signed session token.
- `User` → audit row's `entityId = row.id` (User.id is a CUID, safe).
- All other audited models (`Medication`, `CareUnitMedication`, `Order`, `OrderLine`) → audit row's `entityId = row.id`.

This branching MUST be centralized in `resolveEntityId(modelName, row)` (exported from `auditAllowlist.ts`) and called from `auditExtension.ts` for every audited write. The leak path closed here: returning `row.id` for Session would put a valid session token into `entityId`, surfacing through `/api/audit/events` → admin UI → also visible to anyone reading `audit_events` directly.

  //
  // SECURITY: Session.id IS the raw signed session token. If we naively wrote
  // row.id for Session writes, every auth.login/auth.logout audit row would

  // allowlist closes the `after` column leak; resolveEntityId closes the
  // `entityId` column leak. Both layers must hold for D-97 to be satisfied.
  export function resolveEntityId(model: AuditedModel, row: Record<string, unknown>): string {

      if (typeof userId !== 'string' || userId.length === 0) {
