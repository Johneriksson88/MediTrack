## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL query params → React state | Browser-supplied filter params reach the page; useSearchParams reads them; useMedicationsQuery serializes back to API. |
| LakemedelFilter → URL | Component emits patches; LakemedelPage writes to URL; the URL is the only source of truth. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09 | Tampering | URL query string injection | mitigate | `medicationListQuery` Zod schema validates `q`, `atc`, `form`, `belowThreshold`, `page`, `pageSize` on the BE (already in Plan 01). FE never builds raw SQL; useSearchParams returns strings, Prisma uses parameterized queries. |
| T-02-10 | Denial of Service | High-frequency keystroke search firing on every char | mitigate | 200 ms debounce in LakemedelFilter (UI-SPEC §8a + D-discretion in CONTEXT.md `<decisions>`) — caps client→API request rate; TanStack Query's `placeholderData: keepPreviousData` smooths the pagination flicker. |
| T-02-11 | Information Disclosure | Filter values logged in browser history | accept | Filter values (drug name search terms) are not sensitive on their own; users searching their own vårdenhet's catalog leaks no cross-tenant information. Risk accepted — usability win of deep-linkable URLs outweighs the audit-trail cost. |

Plan 02 mitigates no NEW high-severity threats beyond what Plan 01 established at the API layer. The BE Zod query schema is the canonical guard against malformed/malicious filter params; this plan only wires the UI to those existing guarded routes.
