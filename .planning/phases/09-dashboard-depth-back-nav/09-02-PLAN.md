## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-04 | Information Disclosure | Cross-vårdenhet data leak via the new endpoint | mitigate | Service signature is `careUnitId`-first per D-16; every `prisma.order.findMany` and `prisma.order.count` includes `where: { careUnitId, ... }`. The route hands in `req.user!.careUnitId` from the authenticated session (decorated by `requireSession`). Test 3 asserts isolation — second-vårdenhet rows never appear in the first user's response. |
| T-09-05 | Spoofing / Confused Deputy | Wrong subview shape returned for the wrong role (e.g., apotekare-shape data sent to a nurse session) | mitigate | The service branches on `role` parameter, and the Fastify response schema is `dashboardOrdersResponse` (Zod discriminated union on `role`). A service bug returning the wrong shape would fail serialization at the Fastify layer; Tests 1 and 2 explicitly assert the discriminator matches the session's role. |
| T-09-06 | Spoofing | Client passes a fake role in a header/cookie to elicit the apotekare subview | mitigate | The role is read from `req.user!.role` (decorated by `requireSession` from the DB-loaded User row), NOT from any client-controlled header or query param. No client-supplied input flows into the role branch. |
| T-09-07 | Denial of Service | Unauthenticated requests hammering the endpoint | accept | `requireSession` rejects unauthenticated requests with 401 cheaply (no DB query for orders runs). Authenticated session abuse is bounded by the per-careUnit row count and the `take: 5` cap. No extra rate limit beyond the Phase 5 `@fastify/rate-limit` for `/api/auth/login` (D-122) — read-only dashboard endpoints inherit the default. |
| T-09-08 | Tampering | Cross-tenant injection via Prisma raw query | mitigate | This service uses Prisma's typed query builder, not `$queryRaw`. No string interpolation; `careUnitId`/`userId` are passed as parameters. |

No `medium` or `high` severities at this layer. All cross-tenant boundaries are enforced by the typed WHERE constraints + the careUnitId-first signature; Test 3 mechanically asserts isolation.
